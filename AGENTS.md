# AGENTS.md

## 适用范围

本文件适用于整个仓库，供 AI 编程工具在新增、修改或检查内容时使用。

这个仓库用于实验室成员共享暑期集训 Notebook。目标是让每份内容都能直接阅读、按顺序运行并复现实验结果。它不是个人周计划、周报、LaTeX 报告或其他个人交付物的存放仓库，除非用户明确要求，否则不要把这些内容迁入本仓库。

仓库规范的来源：

- `README.md`：仓库用途、内容分类和整体目录结构。
- `CONTRIBUTING.md`：文件命名、Notebook 最低内容和提交流程。
- 各分类目录的 `README.md`：该分类的主题范围和提交位置。
- `.github/pull_request_template.md`：Pull Request 检查清单。

如这些文档与用户当前的明确要求冲突，以用户当前要求为准。不要自行改写仓库通用规范来掩盖一次性的例外。

## 目录与文件放置

- `01-jupyter/` 到 `08-git-linux/` 是主要内容分类，`optional/` 存放跨分类或扩展内容。
- `templates/` 存放空白模板，各分类下的 `example/` 存放仓库维护的示例，不属于任何成员的个人提交。
- 成员内容必须放在最匹配的分类目录下，并使用自己的姓名文件夹，例如 `04-sklearn/zhangsan/`。
- 同一成员在同一分类中只使用一个个人文件夹，多个相关 Notebook 放在该文件夹内。
- 不要把个人 Notebook 直接放在分类根目录，也不要放进 `example/`。
- 跨分类主题放入最主要的分类，并在 Notebook 首页写明相关分类和关键词。
- 只有任务明确要求维护示例时才修改 `example/` 中的文件。

Notebook 文件名采用：

```text
weekX_topic_name.ipynb
```

其中 `X` 是周次数字，例如 `week2_pipeline_zhangsan.ipynb`。文件名和个人目录建议使用姓名拼音或英文小写，单词之间使用下划线。Notebook 首页中的作者字段填写真实姓名。

## Notebook 最低内容

每个 Notebook 至少应包含：

- 标题、作者、日期、适用周次、分类和关键词。
- 学习目标、必要的前置知识和方法背景。
- Python 版本、主要依赖、数据来源和运行方式。
- 最小可运行示例，以及关键代码前后的 Markdown 解释。
- 实验结果、图表说明、常见错误或注意事项。
- 总结和参考资料链接。

代码必须能按顺序执行，不依赖隐藏状态、个人电脑上的绝对路径或未提交的本地文件。图表应有标题、坐标轴、图例或必要的文字说明。保留有助于阅读的输出，删除调试信息和无关的大型输出。

目录中的示例只展示 Notebook 的组织方式和部分学习内容，不代表完整作业，也不是唯一答案。

如任务要求对照《实验室新生暑期居家集训学习计划》，应以用户提供或仓库可访问的原文为准，不能根据示例反推或编造完整要求。

## AI 工作约束

- 只修改当前任务涉及的文件，不顺手重构、重命名或格式化无关内容。
- 保留成员已有内容和未提交改动，不覆盖、不回退、不删除来源不明的文件。
- 不得伪造运行结果、实验指标、图表、数据来源、参考文献或完成状态。
- 如果无法运行 Notebook，应明确说明未运行的原因、缺少的环境或依赖，不能声称验证通过。
- 编辑 `.ipynb` 时保持合法的 Notebook JSON、有效且不重复的 Cell ID，并尽量避免无关的 metadata 和输出变化。
- 不写入账号、Token、Cookie、SSH 私钥、个人隐私或本地绝对路径。
- 不提交虚拟环境、`.ipynb_checkpoints`、缓存、大型数据集、模型权重或自动生成的临时文件。
- 不为了让示例通过而静默下载大型资源、修改全局环境或降低原有检查标准。
- 未经用户明确要求，不创建 commit、不 push、不合并 Pull Request，也不修改仓库可见性和权限设置。

## 验证要求

修改 Notebook 后应进行与改动范围相称的验证：

1. 使用正确的 Python/Jupyter 内核执行 Restart Kernel and Run All。
2. 确认所有代码单元执行完成，错误输出数量为 0。
3. 验证 Notebook JSON、Cell ID、相对路径和必要输出。
4. 检查 `git diff --check` 和 `git status --short`，确认没有夹带无关文件。

仅修改 Markdown 等文档时，至少检查链接、路径、命名示例以及与 `README.md`、`CONTRIBUTING.md` 的一致性。

## Git 与提交

- 默认使用个人分支和 Pull Request；只有用户明确要求时才直接提交到 `main`。
- 工作区干净且可以快进时，同步最新 `main` 后再开始修改；如存在未提交改动，不得擅自 stash、reset、回退或删除，应保留这些改动并根据任务范围继续处理或向用户说明。
- 提交时只暂存本次任务涉及的文件。
- 一次提交尽量只包含一个主题或一组紧密相关的 Notebook。
- Notebook 提交信息建议使用 `Add <topic> notebook` 或 `Update <topic> notebook`。
- 文档和仓库维护提交使用简洁、能说明实际变更的提交信息。
- 不使用强制推送，不改写其他成员的提交历史。

完成任务时，向用户说明修改了哪些文件、执行了哪些验证，以及仍然存在的限制或未运行项目。
